## New findings to weigh in on (surfaced by the real run)

- [ ] **N1 — Contact-store growth inflates every packet (T1 amplification).** — **Medium/High** —
  `evidence.condition_records` + `tiering`. After bootstrap seeded 54 contacts, the March packet jumped
  to **334 threads / 1.69 MB** (vs Feb's 149 / 0.75 MB) because any thread touching a known address is
  T1 (always kept), bypassing bulk-drop. This grows monotonically as contacts accumulate → unbounded
  packet growth + cost for the future `ApiReasoner`. Question for review: cap/scope the packet (e.g.
  only T1 threads with *activity in the report window*, or relevance-rank within T1), tighten what
  `_seed_contacts` admits (it currently adds *every* human address in surviving threads as role
  "other"), or a hybrid.
- [ ] **N2 — KeepAll surfaces a large volume of (mostly Hebrew) marketing/newsletter threads as T2.** —
  **Medium** — `relevance.KeepAllHumanJudge`. ~200 of March's "non-noise" threads were still marketing
  (sharonramlaor, ku-mi, delitlv, epoch/sherut, wizzair, dominos, …). KeepAll is deliberate (never drop
  a new lead), but the model must wade through heavy noise each run. Question: a *conservative*
  marketing/no-reply heuristic that only demotes high-confidence bulk (List-Unsubscribe + known
  marketing ESP domains), surfacing what it drops — without risking real first-contact leads.
- [ ] **N3 — `_seed_contacts` role granularity.** — **Low** — everything seeds as role "other", so the
  packet's contact roles under-inform the model (it can't cheaply tell an agent from a sub). The model
  *can* infer from threads, but seeding richer roles (or letting insights upgrade roles) would help.
- **Confirmed working well:** recurring-retainer capture (Miracor), human-override persistence, knowledge
  feedback loop, single-use session output, accumulate semantics.
