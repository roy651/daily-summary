## Reviewer notes (external review — static)

Reviewed at the tree above as a standalone checkout. The suite was **not executed** here: `mail-evidence`
is a local editable path-dep (`../private/invoicing-assistant/...`) absent from a standalone clone, so
the golden/integration tests can't run without the sibling. Findings are from static reading of all 17
core modules + the test suite. The architecture is clean, the invariants are well-chosen, and the docs
are unusually disciplined. The findings below are mostly *seam/lifecycle gaps that the offline golden
test structurally cannot catch* (it uses `ReplayReasoner` and a single in-memory run), not defects in
the deterministic core, which is solid.

The scoped items check out: the `apply` no-confirmed-write guard holds (structurally — see F10), the
project-dup fallback exists (but see F5), prioritization/render are deterministic, and the outbound
allowlist raises on a non-match (but see F9). The most important issue (F1) is in the **default**
`REASONER=session` daily path.

## Findings

Severity: **High** = can silently lose/ corrupt data in normal operation · **Medium** = wrong output
or degradation under realistic conditions · **Low** = hardening / hygiene / consistency.

- [ ] **F1 — `SessionReasoner` reuses a stale `model_output.json`; in the default daily path this can
  silently re-apply yesterday's reasoning and (live mode) advance the watermark past unreasoned mail.**
  — **High** — `reasoner.py:46-57`, `cli.py:135-161`, `daily.py:74`.
  `work_dir == state_dir`, so the output path is the fixed `state/model_output.json`. `reason()` returns
  that file whenever it *exists* — and nothing in `daily.py`/`cli.py` ever deletes or rotates it. Day-2
  flow: `daily` sees the day-1 file still on disk → returns day-1 `ModelOutput`, **never writes the new
  packet**, applies stale updates to freshly-loaded state, `FileDelivery` reports `sent=True`, and the
  live-pull `watermark_commit()` fires (`cli.py:160`) — so day-2's pulled mail is now behind the watermark
  but was never reasoned. This is precisely the stated worst-case ("losing an important email"). The golden
  test can't see it (it uses `ReplayReasoner` + one run). *Fix:* make the session output single-use — after a
  successful load, rename to `model_output.<run_date>.json` (or require/scope paths per `run_date` so a prior
  day's file is never mistaken for the current one), and don't advance the watermark unless a packet was
  actually produced this run.

- [ ] **F2 — `done`/`archived` projects are never filtered out of the reasoning packet; `current_projects`
  grows monotonically and invites re-surfacing of closed work.** — **Medium** — `daily.py:64-73`,
  `packet.py:125`. The doc says the model "always sees the full current state of every *open* project," but
  the code feeds every project regardless of status, plus each one's full (append-only, never-pruned)
  `evidence_thread_ids` (`packet.py:60`, `apply.py:66-70`). Over months the packet bloats and a `done`
  project can be re-activated by the model each run. *Fix:* drop `status in {done, archived}` from
  `current_projects` (keep them in state), and cap/trim per-project `evidence_thread_ids` fed to the packet.

- [ ] **F3 — `deadline_kind` is never validated against an enum anywhere; a typo silently downgrades a hard
  deadline to soft weighting.** — **Medium** — `state.py:175` (comment-only `"hard"|"soft"|None`, no
  `_check`), `schema.py:44-73` (validates `status_agent`/`confidence` but not `deadline_kind`),
  `todos.py:73`. Prioritization does `weight = 1.0 if deadline_kind == "hard" else 0.4`, so any non-exact
  value (`"Hard"`, `"firm"`) silently scores as soft — a missed-deadline risk that contradicts the
  "reject unknown enums → fail loudly" contract. *Fix:* add a `DEADLINE_KINDS` frozenset and `_check` it in
  both `Project.__post_init__` and `ProjectUpdate.from_dict`.

- [ ] **F4 — Task-level prioritization coerces any `due_hint` into a *hard* deadline, asymmetric with the
  identical project-level path.** — **Medium** — `todos.py:127-131` vs `103-109`. For task-level todos,
  `kind = p.deadline_kind if (todo.due_hint or task.deadline) is None else "hard"` — so a todo carrying a
  soft model-inferred `due_hint` gets full weight and can trip `hard_soon → urgent`. The *same* `due_hint`
  on a project-level todo uses `p.deadline_kind` (soft). A `due_hint` is the model's soft guess; treating it
  as hard is both a bug and an inconsistency that makes the band depend on whether a todo happens to hang off
  a task. *Fix:* use `p.deadline_kind` uniformly; never synthesize `"hard"` from the presence of a hint.

- [ ] **F5 — The project-dedup fallback can *wrongly merge* two distinct same-client projects, silently
  clobbering the matched project's agent fields.** — **Medium** — `apply.py:30-47`. Token-overlap ≥ 0.6 with
  `min(len(want), len(have))` in the denominator means a subset title matches fully: `"brochure"` vs
  `"brochure design"` → 1.0; `"logo"` vs `"logo refresh"` → 1.0. On a wrong match `_apply_one` overwrites
  `status_agent`/`status_evidence`/`confidence`/`blockers`/`deadline` and merges todos into the wrong
  project. The design explicitly prefers *duplication* over collision ("recall is the gate"), but this
  heuristic risks the unsafe direction. *Fix:* raise the threshold and/or require an additional signal
  (shared evidence thread id, exact normalized-title); when ambiguous, prefer creating a new project and
  surface it for human dedup rather than auto-merging.

- [ ] **F6 — Blockers can never be cleared by the agent, and `blocks_until` is never consulted; stale
  blockers persist indefinitely, inflating priority and misleading the digest.** — **Medium** —
  `apply.py:71-72` (`if update.blockers:` — an empty list is a no-op, so the model can't resolve a blocker),
  `todos.py:79-80` (+15 leverage while `status in {blocked,on_hold}`), `state.py:84` (`blocks_until` unused).
  Unlike todos, blockers have no surfaced "carry vs. resolve" story and no auto-expiry. *Fix:* let the model
  replace blockers with an explicit empty list (distinguish "omitted" from "[]" in the schema), and/or
  de-escalate/auto-clear blockers whose `blocks_until` is past the run date.

- [ ] **F7 — `daily --as-of <date>` does a *live IMAP pull* unless `--from-export` is given, so a "replay a
  past day" backtest can hit the live mailbox and advance the real watermark.** — **Medium** —
  `cli.py:126-133`, `RUNBOOK.md` ("replay a held-out day" shows `--as-of` with no export source). The GT
  backtest in `docs/07` depends on deterministic replay; this footgun makes it stateful. *Fix:* require an
  offline source (`--from-export` or a fixtures dir) whenever `--as-of` is in the past, or refuse to commit
  the watermark on an `--as-of` run.

- [ ] **F8 — `packet.json` (full email bodies, addresses, attachments metadata) is written to `state/` in
  cleartext on every session run and never cleaned up.** — **Low/Medium** — `reasoner.py:51-54`,
  `packet.py:88-106`. Git-ignored (good), but for a privacy-first product a growing at-rest cache of raw
  correspondence (plus the lingering `model_output.json` from F1) is a real footprint. *Fix:* treat
  `packet.json`/`model_output.json` as ephemeral scratch (rotate or delete after a successful run), or write
  them under a `tmp/` that the runbook documents as transient.

- [ ] **F9 — The outbound "Avigail-only" invariant enforces `recipient == DIGEST_EMAIL_TO`, not
  `recipient == Avigail`; a single env typo would send the whole digest (client correspondence summaries)
  to the wrong address with no secondary check.** — **Low** — `delivery.py:60-82`, `cli.py:124`. The
  allowlist-of-one is the right shape, but correctness rests entirely on env hygiene. The `override_to`
  parameter is also effectively dead (it can only equal `self.to` or raise) — a small smell. No SMTP timeout
  on `SMTP_SSL` (`delivery.py:104`) can hang a scheduled run. *Fix:* document that `DIGEST_EMAIL_TO` is
  safety-critical; consider a confirmed-recipient file checked into `state/` rather than a bare env var; add
  an SMTP timeout; drop or justify `override_to`.

- [ ] **F10 — The no-confirmed-write guard is structural + test-only; there's no runtime assertion in
  `apply` itself, and the existing test starts from `status_confirmed = None`.** — **Low** —
  `apply.py:57-114`, `test_apply.py:52-58`. Today the guard holds because the write code simply never names
  `status_confirmed`/`confirmed_note` — but the only thing stopping a future regression is a test that
  checks the field *stayed None*, not that apply *preserved a pre-existing confirmed value*. *Fix (cheap
  insurance):* snapshot `(status_confirmed, confirmed_note)` per matched project at the top of
  `apply_model_output` and `assert` they're unchanged at the end; add a test where confirmed is pre-set and
  the model tries to change it.

- [ ] **F11 — Inconsistent failure mode for missing required fields: `KeyError` vs the loud `ValueError`
  used elsewhere.** — **Low** — `schema.py:88` (`d["headline"]`), `schema.py:102` (`d["thread_id"]`),
  `state.py` `from_dict`s. The contract is "missing required fields → fail loudly"; a `KeyError` is loud but
  opaque (no field name / context) and bypasses the deliberate `_check` style. *Fix:* validate required keys
  explicitly with the same `ValueError` message shape.

- [ ] **F12 — Minor doc drift / reproducibility.** — **Low** — `cli.py:5` docstring lists subcommands
  without `score` (it exists, `cli.py:287`). Separately, the suite is not runnable from a standalone clone
  (sibling path-dep); a CI note or a published `mail-evidence` test stub would let an external reviewer
  actually run the golden gate. *Fix:* update the docstring; document the sibling checkout requirement (or a
  stub) for reviewers/CI.

### Not findings — confirmed sound
- `apply` reads-but-never-writes the confirmed columns; effective `status` precedence is correct
  (`apply.py:85`, `test_apply.py:60-71`).
- `last_activity_date` is recomputed from evidence and is monotonic / never regresses
  (`apply.py:90-95`).
- `is_self_generated` drops our own outbound digest but correctly keeps a `Re: digest:` reply as inbound
  feedback (`relevance.py:25-31`).
- Prioritization sort + render ordering are fully deterministic (stable tie-breaks on
  `project_id`/`text`; `todos.py:148`, `render.py:41-46`). Note: digest-update order within an importance
  band follows model output order (golden-stable via fixture; acceptable as editorial order).
- State JSON round-trips deterministically (field-order dumps, `None↔null`; `state.py:264-272`).
